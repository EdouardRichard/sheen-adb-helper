# ADR 0006：Logcat 最近历史与持续跟随模式

- 状态：`accepted`（v0.03 设计已接受，尚未实现）
- 日期：2026-07-18
- 负责人：Sheen ADB 助手项目
- 当前版本决定：v0.02 不修改 Logcat 实现

## 背景

v0.02 真机回归确认当前 Logcat 使用持续 `shell-v2` 流且资源清理正确，但启动后会从设备缓冲区较早位置持续读取。缓冲区历史较多时，界面即使只展示最新 100 条，也可能需要较长时间才能追到设备当前日志尾部。

Logcat 的主要用户目标是：应用发生异常后，用户能够尽快看到相关的最近日志，自行分析或通过明确的复制/导出动作提交给开发者；不是观看从缓冲区起点开始的长时间回放。

该问题涉及 Logcat 命令策略、ROM 能力探测、初始历史与持续流的无缝交接、重复/缺失控制和日志隐私，属于生命周期规范要求记录的日志策略变更。本 ADR 只登记 v0.03 目标，不授权在 v0.02 中修改代码。

## 候选方案

1. 保持现状，仅依靠界面 100 条窗口：实现简单，但到达当前日志尾部过慢，不满足异常分析目标。
2. 启动时只跟随新日志：最快，但用户在打开页面前已经发生的异常可能完全丢失。
3. 先读取有界的最近历史，再从同一时间游标持续跟随，并处理交接处重复：兼顾启动速度和异常前后文；采用为 v0.03 目标方案。
4. 自动后台常驻采集：可保留更多历史，但违反前台、用户主动开始和不后台采集的不可变约束；不采用。

## 决策

v0.03 采用以下设计；实现仍须由已批准的 `tasks.md` 授权：

- Logcat 默认启动模式调整为“有界最近历史 + 持续跟随”，进入页面后应快速到达设备当前日志尾部，同时保留一段可用于异常分析的最近上下文。
- 默认最近历史上限为开始游标前 5 分钟、设备端最近 512 条、Core 内存 1 MiB，单个解码行不超过 64 KiB；历史/能力阶段预算 2 秒，从点击开始到历史、空历史或降级结果的总预算为 3 秒。持续 follow 不设置空闲读超时，但仍受页面、用户、Session 和资源关闭生命周期约束。
- Core 从被控端取得严格纯数字 epoch 游标，使用相同 buffer、等级和可解析格式完成有界 dump，再从同一游标 follow；向 Feature 交付项目自有结构化事件并保持既有 threadtime 展示语义。
- 交接去重使用最多 512 项的 exact-record fingerprint 多重集，只抑制已确认的边界重放次数；不得用宽泛时间窗口或纯文本去重。
- 所有 dump、时间游标、持续流命令、ROM 能力探测、降级、超时、取消和资源清理仍只位于 `:core:adb`。
- 必须以实际行为探测并验证不同 Android/ROM 对设备游标、count tail、time tail、缓冲区选择及输出格式的支持；不支持默认模式时结束当前 flow 并等待用户明确选择“仅跟随新日志”或“只读取有界最近历史”，不能自动选择或静默回退成长时间全缓冲回放。
- 文件传输/APK 提取与 Logcat 使用 Session-scoped 原子独占租约；冲突只提示用户先停止或取消现有任务，不自动替换任务。文件浏览和短查询不占用该租约。
- Feature 继续只接收结构化日志行，展示过滤后的最新 100 条；不得拼接 Logcat 命令或访问 Socket。
- 仍只在页面前台且用户明确点击开始后采集；离开、停止、取消、断开或切换 Session 时立即释放资源。
- 日志仍默认仅在内存中，不自动落盘、不上传、不写入诊断事件；保留用户主动复制或 SAF 导出的既有能力。

## 影响

- v0.02 仅记录该已知体验问题，代码、权限、依赖、Manifest 和数据格式均不因本 ADR 改变。
- v0.03 需要新增 core 层命令/能力测试以及真机矩阵，覆盖历史为空、历史较多、持续到达、游标边界、重复、时间跳变、ROM 不支持、超时、取消、断开和 Session 切换。
- 最小矩阵覆盖 Android 11/API 30、Android 13/API 33、至少两类厂商 ROM 和 Android 16/API 36；证据只记录合成标记计数、时延和结构化结果。
- 需要以异常发生在“打开页面前”和“开始采集后”两类场景分别验收，且验收记录不得包含真实 Logcat。

## 回滚方式

若 v0.03 的游标能力在目标 ROM 上无法可靠实现，回滚为显式、可选择的“仅跟随新日志”与“读取有界最近历史”两种模式；不得回滚到无界或长时间追赶的隐式行为，也不得引入后台采集。
